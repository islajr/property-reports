# Codebase Reference Manual — Nigerian Real Estate Market Reports

This document serves as the comprehensive developer manual for the Nigerian Residential Real Estate Market Reports platform. It explains the system architecture, database schemas, Java backend components, Python compilation scripts, rendering engine, and dashboard interfaces.

---

## 1. System Architecture

The platform is split into two independent codebases:

1. **`property-intel` (Java API Backend)**:
   * **Role**: Manages the ingestion pipeline and database compiling.
   * **Stack**: Spring Boot, Java 21, Maven, Flyway, Supabase (PostgreSQL).
   * **Responsibility**: Scrapes raw listing data, cleans it, compiles weekly neighbourhood snapshots, and runs the Flyway migrations.
2. **`property-reports` (Python Compilation Pipeline)**:
   * **Role**: Builds the static publishing directories and PDFs.
   * **Stack**: Python 3.10+, Pandas, Jinja2, WeasyPrint, Google Generative AI (Gemini SDK).
   * **Responsibility**: Computes metrics, calls Gemini for narratives, compiles HTML reports with scoped CSS variable themes, regenerates the index page, and publishes robots/sitemaps.

## 2. Database Schema & Data Ingestion

Listing data is compiled in a PostgreSQL database hosted on Supabase.

### Ingestion Tables
* **`raw_data.scraped_listings`**: The raw listing dump. Contains fields like `title`, `price_kobo`, `city`, `neighbourhood`, `price_type` (values: `FOR_SALE`, `FOR_RENT`, `FOR_SHORT_LET`), and `scraped_date`.
* **`market.neighbourhood_snapshots`**: The compiled analytics table. Aggregates listings by neighbourhood, city, snapshot week, and price type.

### Snapshot Primary Key & Hash Resolution
To avoid duplicate snapshots and hash collisions, the primary key of `market.neighbourhood_snapshots` is a **128-bit MD5 hash** generated by concatenating:
`city` + `neighbourhood` + `snapshot_week` + `price_type`

```sql
md5(concat(lower(city), '|', lower(neighbourhood), '|', snapshot_week::text, '|', price_type))
```

### Database Migration: `V8__split_snapshots_by_price_type.sql`
This migration Restructured the historical snapshots table:
1. Renamed the old combined table to `market.neighbourhood_snapshots_old`.
2. Created the new `market.neighbourhood_snapshots` table containing the `price_type` column.
3. Extracted 13 weeks of history (`2026-04-06` to `2026-07-06`) by running a grouping select query over `raw_data.scraped_listings` and grouped by `price_type` (excluding `FOR_SHORT_LET`).

## 3. Java Backend Components (`property-intel`)

Located in the `property-intel` repository.

### JPA Entity: `Market.java`
Mapped to `market.neighbourhood_snapshots`, tracking active metrics per price type:
```java
@Entity
@Table(name = "neighbourhood_snapshots", schema = "market")
public class Market {
    @Id
    private String id; // MD5 hash key
    
    private String city;
    private String neighbourhood;
    
    @Column(name = "snapshot_week")
    private LocalDate snapshotWeek;
    
    @Column(name = "price_type")
    private String priceType; // "FOR_SALE" or "FOR_RENT"

    @Column(name = "active_listing_count")
    private int activeListingCount;

    @Column(name = "new_listings_count")
    private int newListingsCount;

    @Column(name = "price_reduced_count")
    private int priceReducedCount;

    @Column(name = "median_price_kobo")
    private Long medianPriceKobo;
    
    @Column(name = "avg_days_on_market")
    private Double avgDaysOnMarket;
}
```

### Repository: `MarketRepository.java`
Contains the native database query `refreshMarketSnapshot(LocalDate weekStart, LocalDate weekEnd)` which aggregates listings into weekly snapshots:
* **Active Listing Count**: Sum of scraped listings active during the snapshot week.
* **New Listing Count**: Number of listings first discovered during the snapshot week.
* **Price Reduced Count**: Number of unique listings where the price decreased compared to the previous week's observation.
* **Median Price Kobo**: Aggregated percentile calculations (`percentile_cont(0.5)`) computed inside PostgreSQL.

## 4. Python Pipeline Core Scripts (`property-reports`)

Located in the `property-reports` repository.

### 4.1. Exporter: `export_data.py`
Connects to the Supabase database using PostgreSQL credentials loaded from `.env`.
* **Execution**: Runs a query selecting `snapshot_week`, `city`, `neighbourhood`, `price_type`, `active_listing_count`, `new_listings_count`, `price_reduced_count`, `median_price_kobo`, and `avg_days_on_market`.
* **Output**: Overwrites [data/neighbourhood_snapshots.csv](file:///home/isla-jr/Documents/se-workspace/property-reports/data/neighbourhood_snapshots.csv).

### 4.2. Metrics Engine: `compute_metrics.py`
The statistical core of the reports builder.
* **`load_snapshots(csv_path)`**: Loads snapshots from the CSV. Discards rows where `price_type` is `FOR_SHORT_LET`.
* **`filter_by_period(df, end_month, period)`**: Calculates date boundaries. For a target month (e.g. `2026-06`) and period (e.g. `3mo`), it filters weeks from April 1st to June 30th.
* **`compute_kpis(df)`**: Aggregates peak active listings, sum of new listings, sum of price reductions, and percentage listing growth.
* **`compute_ranking(df)`**: Ranks neighbourhoods by average weekly active listings.
* **`compute_pricing_tiers(df)`**: Groups by neighbourhood and calculates average median price and days on market (DOM). Filters out areas with insufficient listing counts to reduce outlier noise.
* **`compute_weekly_aggregates(df)`**: Groups snapshots by week to chart market trajectory.
* **Output**: Saves `data/metrics_{month}_{period}.json` containing distinct nested `"sales"` and `"rentals"` sections.

### 4.3. Narrative Generator: `generate_narrative.py`
Communicates with the Gemini SDK (`gemini-1.5-flash`).
* **Existing Narratives Check**: Checks if a narrative file already exists. If found, it skips execution to **prevent overwriting published content**.
* **Structured Prompts**: Passes computed metrics to Gemini along with a schema prompt.
* **Pydantic Response Schema**: Resolves Gemini responses into structured JSON blocks:
  ```json
  {
    "executive_summary": "...",
    "key_takeaways": ["Point 1", "Point 2"],
    "inventory_narrative": "...",
    "pricing_narrative": "...",
    "lagos_narrative": "...",
    "abuja_narrative": "...",
    "other_cities_narrative": "...",
    "price_reduction_narrative": "...",
    "outlook": "..."
  }
  ```
* **Output**: Saves a dual-keyed narrative structure: `{"sales": {...}, "rentals": {...}}` to `data/narrative_{month}_{period}.json`.

### 4.4. Report Renderer: `render_report.py`
Compiles reports and maintains subdomain assets.
* **`render_html(template_vars)`**: Invokes Jinja2 to compile the template.
* **`compile_pdf(html_path, pdf_path)`**: Calls WeasyPrint programmatically to render A4 PDF files.
* **`regenerate_index(reports_dir)`**: Scans report subdirectories, parses `meta.json` records, and compiles the root dashboard listing.
* **`generate_sitemap_and_robots(reports_dir)`**: Automatically outputs `robots.txt` and `sitemap.xml` mapping all published endpoints.

## 5. Templating, Styling & Visual Themes

Reports are compiled using a single template: [templates/report.html.j2](file:///home/isla-jr/Documents/se-workspace/property-reports/templates/report.html.j2).

### Jinja2 Loop & Suffix Logic
The template uses a DRY Jinja2 loop that iterates over `['sales', 'rentals']`. If metrics exist for the market type, it builds a section with proper headings:
```jinja2
{% for mtype in ['sales', 'rentals'] %}
  {% set mdata = metrics[mtype] %}
  {% if mdata %}
    {% set suffix = "/yr" if mtype == "rentals" else "" %}
    <!-- Pricing tables format: ₦1.5M{{ suffix }} -->
  {% endif %}
{% endfor %}
```

### CSS Scoped Variable Themes
Accent colors are controlled using custom CSS variables scoped to `.period` and `.type` classes:

### Print Media Overrides (`@media print`)
* **Color Adjustments**: Sets `-webkit-print-color-adjust: exact` and `print-color-adjust: exact` to print background cards and chart gradients.
* **Hidden Navigation**: Hides website headers (`.global-header`) and download buttons (`.download-pdf-btn`).
* **Flex Wraps**: Toggles grid layouts to vertical stacks so tables do not overflow page boundaries.
* **Page Breaks**: Uses `page-break-inside: avoid` on sections and cards to prevent orphan headers.

## 6. Interactive Explorer Dashboard

The root directory landing page ([reports/index.html](file:///home/isla-jr/Documents/se-workspace/property-reports/reports/index.html)) contains a client-side search and filtering dashboard.

```
+-----------------------------------------------------------+
|               Market Intelligence Dashboard               |
+-----------------------------------------------------------+
| [ Search reports by month/year...                       ] |
|                                                           |
| Market:      ( All )  ( Sales Only )  ( Rentals Only )    |
| Frequency:   ( All )  ( Monthly )  ( Quarterly )  ( Yearly) |
+-----------------------------------------------------------+
|  +--------------------+   +----------------------------+  |
|  | June 2026          |   | June 2026 (3-Month)        |  |
|  | [Monthly] [Sales]  |   | [Quarterly] [Combined]     |  |
|  +--------------------+   +----------------------------+  |
+-----------------------------------------------------------+
```

### 1. Data Serialization
All published reports are compiled into a JSON array embedded at the foot of the document:
```html
<script id="reports-data" type="application/json">
  [
    {"month": "2026-06", "label": "June 2026", "period": "1mo", "type": "all", ...},
    {"month": "2026-06_3mo", "label": "June 2026 (3-Month)", "period": "3mo", "type": "all", ...}
  ]
</script>
```

### 2. Live Client-Side Filtering
A Vanilla JS script reads the JSON array, monitors the search field and filter pills, and re-renders report cards dynamically:
* **Search Match**: Matches the report label or folder date.
* **Type Match**: Filters by transaction category (`all`, `sale`, `rent`).
* **Period Match**: Filters by reporting window (`all`, `1mo`, `3mo`, `6mo`, `12mo`).
* **Show More Pagination**: Displays 6 reports initially and increments by 6 when clicking the "Show More" button.

### 3. SEO Fallback
If Javascript is disabled, the page falls back to server-side Jinja2 lists of HTML links, allowing web crawlers (like Googlebot) to discover and index all reports.

## 7. Known Issues & Design Constraints

### 7.1. Outage Handling (Carried Forward Listings)
* **Problem**: Network outages or portal blocks occasionally cause scraper runs to fail.
* **Solution**: In weeks with a scraped count of zero, the pipeline carries forward active listing counts from the prior week to preserve historical continuity. New listing counts are set to zero for that period.

### 7.2. Days on Market (DOM) Truncation
* **Problem**: Days on market is calculated as the date difference between a listing's first scraping discovery and the snapshot date.
* **Constraint**: Listings that were already active before the scraping pipeline launched will show days on market truncated to the pipeline's inception date, skewing DOM statistics downwards in the first few months of historical tracking.

### 7.3. Short Let Pricing Exclusion
* **Problem**: listings marked `FOR_SHORT_LET` contain varying payment structures (e.g., daily rates vs. weekly or monthly rates) that portals fail to standardize.
* **Constraint**: Short let listings are completely excluded from both metric calculation and sitemap compilation to prevent extreme outliers from skewing rental pricing averages.
